# Judge Report: speech-to-spell

**Track:** Track 1 — Anything Goes / AWS
**Repository:** https://github.com/charles-azam/speech-to-spell

## Summary

Speech to Spell is a real-time multiplayer wizard duel game where players speak spells out loud — Voxtral Mini transcribes the voice, Ministral 8B judges the spell's creativity and coherence (assigning damage and visual effects), and a second Ministral 8B instance drives a live sports-commentary duo that reacts after every turn. The project is end-to-end functional, deployed on EC2 + Cloudflare Pages, and exhibits a latency-first design philosophy that is unusually deliberate for a 48-hour hackathon. The game concept is immediately fun and produces genuinely entertaining moments.

## Strengths

- Latency-first architecture is explicitly designed, documented, and consistently executed: Voxtral Mini + Ministral 8B chosen for speed, commentator is fire-and-forget, sound effects are pre-generated and disk-served at runtime, TTS is cached in memory, EC2 is co-located in Europe near Mistral inference endpoints.
- Dual-agent design with clearly separated concerns: Judge Agent blocks the game loop (single tool call, result needed), Commentator Agent is async (multi-tool-call, `marc_says` / `sophie_says`, loops autonomously every 20 seconds of silence).
- Tool-calling is used fluently throughout: `judge_spell` tool with 10 fields (verdict, comment, spell_name, target, damage, sound_id, emojis, template, primary_color, secondary_color), `select_emojis` for emoji inference, `marc_says` / `sophie_says` for the commentator duo — all with tight JSON schemas and server-side validation.
- Frontend is genuinely polished: 9 particle animation templates in `SpellEffect.tsx` (explosion, swirl, rain, wave_left, wave_right, shatter, pulse, spiral, rise), audio ducking via Web Audio API `GainNode` when recording, exponential backoff WebSocket reconnection, screen shake for big damage, push-to-talk with per-player microphone selection, bilingual i18n.
- Code quality is high for a 48-hour project: clean module separation, Pydantic models throughout, lazy singletons with global client reuse, error handling for dead WebSockets, race condition fix for concurrent pending explanations (dict keyed by player side), FIFO cache eviction.
- The `EXPLAIN` verdict mechanic is a creative design touch — it produces a moment of theatrical tension before the judge delivers a final verdict.
- Deployment is complete and production-ready: Docker + docker-compose + nginx reverse proxy with WebSocket upgrade headers, Cloudflare Pages for the frontend, password-gated to conserve API credits.

## Areas for Improvement

- The AWS Bedrock integration (`_get_aws_client()`, `AWS_MODEL`, `SPELL_PROVIDER = "mistral"`) is present as dead code. If this is the claimed Track 1 differentiator, it should be demonstrated, load-balanced, or at least toggled via environment variable at demo time.
- Room state is held entirely in-process Python dicts (`_rooms`, `_room_websockets`). A single server crash loses all active games with no recovery path. For a hackathon this is acceptable, but it means the system cannot scale beyond a single instance.
- The commentator idle loop fires every 20 seconds but the `judge_busy_until` check and TTS generation happen sequentially inside the loop — if ElevenLabs is slow, commentary lines can pile up or block the next commentary cycle.
- No tests are present anywhere (backend or frontend). Given the non-trivial game logic (consume-and-refill, EXPLAIN flow, broadcast filtering), even a handful of unit tests for `game.py` and `spell.py` would meaningfully improve confidence.
- The `progress.md` file references RAG (Mistral Embed + Qdrant for sound/image lookup) and VAD (Silero) as not yet implemented. These would have pushed the technicity score higher.
- The `infer_emojis()` function fallback (`valid = hand[:1]`) silently picks one emoji when inference fails — this should arguably broadcast a warning or fizzle the spell rather than silently degrading.
- Hand size is fixed at 10 emojis with no scarcity — the emoji hand is a creative scaffold but never creates strategic tension. Introducing limited or rare emojis could deepen gameplay.

## Detailed Reasoning

**Architecture and Mistral integration.** The pipeline in `voice.py` is clean: audio bytes arrive over WebSocket, are forwarded to `mistral.audio.transcriptions.complete(model="voxtral-mini-latest")` with a retry wrapper for transient network errors, and the text result is broadcast back to the room immediately as a `transcription` message. The judge in `spell.py` makes a single structured `chat.complete()` call with `tool_choice="any"` and the `judge_spell` tool, parsing the result via `_parse_judge_tool()`. Damage scaling from the tool output (`particle_count = 15 + int(damage * 1.17)`, `scale = 0.6 + damage / 30 * 1.4`) ensures visual effects are proportional to game impact without requiring the LLM to think about rendering. The commentator in `commentator.py` fires after each spell in a background `asyncio.create_task`, waits for `room.judge_busy_until` before speaking so voices never overlap, and generates dialogue via multi-tool-call — the model chooses how many times to invoke `marc_says` and `sophie_says`, producing 1 to 3 lines of natural back-and-forth. The system prompt for both agents is carefully tuned: the judge has explicit damage bands (1-5 for lazy spells, 8-15 for decent, 20-30 for original), a hardcoded 75%/20%/rare split for YES/NO/EXPLAIN, and a detailed French argot personality with example comments.

**Game logic and state management.** `game.py` is clean and well-separated from the WebSocket layer. `deal_hand()` samples without replacement from a 200+ emoji bank. `consume_and_refill()` removes used emojis and refills without duplicating against the remaining hand. `apply_spell()` handles both attack and heal, caps health at MAX_HEALTH, and sets a winner when HP reaches zero. `format_game_context()` feeds actual wizard names and recent spell history to the judge prompt, giving the LLM enough context to penalize repetition. The `pending_explanations` dict (keyed by player side) correctly handles the edge case where both players receive EXPLAIN simultaneously, which is a subtle correctness fix visible in `progress.md` and confirmed in `room.py`.

**Frontend quality.** The React frontend uses a `useReducer`-based state machine in `useGameState.ts` with explicit action types (`server_message`, `set_processing`, `clear_screen_shake`, `clear_visual_effect`, `clear_spell_state`, `connection_lost`). All side effects (timed cleanup of visual effects and spell state) are scheduled via a tracked timeout ref that is cleared on unmount — this is idiomatic React and avoids stale closure bugs. The WebSocket hook (`useWebSocket.ts`) implements a commentator audio queue with a playing flag, 300ms inter-line gaps, and an Audio API `GainNode` for ducking — commentary is audible but fades when the player is recording. The `SpellEffect.tsx` component generates particle arrays for each of the 9 templates using parameterized geometry (angles, distances, spiral step), then animates them via inline CSS keyframes with CSS custom properties (`--dx`, `--dy`, `--rot`) for per-particle offsets — this is a well-crafted particle system built entirely without a rendering library.

**Deployment and operational readiness.** The Docker setup is complete: `uv` is copied from the official image layer, dependencies are installed from the lockfile, the sound cache is baked into the image, and uvicorn binds to `0.0.0.0:8000`. nginx handles WebSocket upgrades with a 24-hour `proxy_read_timeout`. CORS is configurable via `ALLOWED_ORIGINS`. A password gate protects the API during the post-hackathon demo period. The background `periodic_cleanup` task removes stale rooms every 5 minutes. This level of operational polish is notably above average for a 48-hour build.

**Track 1 / AWS alignment caveat.** The `spell.py` file contains a full AWS Bedrock client (`_get_aws_client()` using `openai.OpenAI` with a Bedrock-mantle endpoint) and a separate `_interpret_openai()` path that mirrors the Mistral path. A `SPELL_PROVIDER` constant controls which path is taken, but it is hardcoded to `"mistral"` at the top of the file. This means the AWS path was tested at some point but is not active at demo time. The project's README does not mention AWS, and the server runs on EC2 but that is infrastructure, not model inference. The Track 1 AWS angle is present in code but not demonstrated, which limits the alignment score.
