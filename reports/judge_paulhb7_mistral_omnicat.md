# Judge Report: paulhb7/mistral_omnicat (OmniCAT)

**Track:** Track 1 "Anything Goes" (AWS)
**Repository:** https://github.com/Paulhb7/mistral_omnicat

## Summary

OmniCAT is an ambitious multi-agent OSINT (Open Source Intelligence) platform that orchestrates six specialist AI agents — covering maritime surveillance, aviation tracking, natural hazards, geopolitical conflicts, solar weather, and exoplanet research — via Mistral models on AWS Bedrock. The system features a real-time voice interface powered by Mistral Voxtral for STT and ElevenLabs for TTS, a cinematic Jarvis-inspired HUD with a Three.js voice-reactive particle orb, and four switchable 2D/3D map views including embedded NASA Eyes viewers. The project is technically dense, well-structured, and delivers an unusually polished demo experience for a 24-hour hackathon.

## Strengths

- **Breadth of live integrations**: Seven distinct external APIs integrated and functional — AISStream (WebSocket), NASA EONET, USGS, ACLED (OAuth bearer), GDELT, NASA DONKI, NASA NeoWs, NASA Exoplanet Archive (TAP/ADQL), arXiv (XML Atom feed), Nominatim, Open-Meteo. Each integration has proper async/await, timeout handling, and graceful degradation on failure.
- **Agent-as-tools architecture**: The pattern in `agent_tools.py` of wrapping each `strands.Agent` instance with a `@tool` decorator so the orchestrator can invoke them as callable tools is elegantly implemented and extensible — the README even documents the five-step process to add a new agent.
- **Two-tier model strategy**: Using Mistral Large for orchestration reasoning and Ministral-3B for specialist agents reflects genuine cost/quality reasoning, not just picking one model for everything.
- **Real-time voice pipeline**: The full listen → Voxtral STT (WebSocket PCM s16le 16kHz) → orchestrator → ElevenLabs TTS → speak → listen loop with barge-in detection (RMS energy threshold + 3-frame confirmation + 500ms cooldown) is working end-to-end. This is a non-trivial real-time audio engineering challenge solved cleanly in `chat/page.tsx`.
- **Three.js OmniOrb**: The 2800-particle + 700-halo Fibonacci sphere with real-time audio frequency deformation via `createMediaElementSource` → `AnalyserNode` is technically impressive and connects the voice interaction to a physical visualization feedback loop.
- **Synthesized HUD sound effects**: `utils/sfx.ts` implements all UI sounds from first principles using Web Audio API oscillators, filtered noise buffers, slapback delays, and stereo detuning. No audio assets required — all sounds are synthesized at runtime.
- **Streaming architecture**: The SSE event typing system (`content`, `tool_start`, `tool_end`, `agent_selected`, `location`, `data_*`) combined with the data bus pattern cleanly separates narrative LLM output from structured tool data, allowing the frontend to update map overlays and intel panels in real time.
- **SQLite maritime persistence**: Maritime AIS data is stored in a local SQLite database with proper schema (vessel_positions, vessel_static, monitored_entities tables, indexed by MMSI and timestamp), allowing position history retrieval without re-querying the live stream.
- **CI pipeline and Docker**: GitHub Actions CI runs backend tests (pytest), frontend build (Next.js), and Docker builds for both services. Dockerfiles for both backend and frontend are present.
- **Documentation quality**: `docs/` contains six detailed markdown files (architecture, agents, data sources, deployment, frontend, voice pipeline). `AGENTS.md` is a complete codebase guide. The orchestrator system prompt is sophisticated — it includes precise disambiguation rules between solar_system_analyst and milky_way_analyst and even instructs the agent to never geocode celestial bodies.

## Areas for Improvement

- **Stale test suite**: `backend/tests/test_orchestrator.py` imports and tests `run_orchestrator` and `_run_agent` functions that no longer exist in `orchestrator.py` (the current implementation uses `stream_orchestrator` async generator instead). These tests would fail against the actual codebase, reducing the CI value.
- **Aviation risk analysis is mocked**: `aviation_tools.py:check_aircraft_risk` has a hardcoded sanctions list of exactly two ICAO addresses (`["ABC123", "DEF456"]`) and `_detect_anomalies` always returns `[]`. This is explicitly a stub, but it is presented as a functional capability in the README.
- **Data bus is not session-safe**: `tools/data_bus.py` uses a module-level list (`_pending: list[dict] = []`) acknowledged in its own docstring to be unsafe for multi-worker or concurrent-session deployments. Under load with multiple simultaneous users, events would bleed across sessions.
- **CORS wildcard**: `server.py` sets `allow_origins=["*"]`, acceptable for a hackathon demo but noted for any production path.
- **Frontend hardcodes localhost**: `use-chat.ts` defaults to `http://localhost:8000/stream` and `chat/page.tsx` hardcodes `ws://localhost:8000/ws/stt` and `http://localhost:8000/tts`. An environment variable or Next.js `NEXT_PUBLIC_API_URL` would be needed for any deployment.
- **No streaming on TTS**: The TTS endpoint in `server.py` uses `httpx` to stream audio bytes from ElevenLabs to the client, but `chat/page.tsx` waits for the full response blob before playback (`await resp.blob()`). True streaming playback (using `MediaSource` API) would reduce time-to-first-audio.
- **Missing milky_way agent test**: `test_agents.py` tests five of the six agent factories but omits a test for `create_milky_way_agent`.

## Detailed Reasoning

**Technicity (23/25):** The codebase demonstrates strong command of async Python, the strands-agents framework, and real-time frontend engineering. The maritime module is the most complete, with a full WebSocket listener, SQLite persistence across three normalized tables, proper MMSI vs. name search branching, and geo-bounded area monitoring. The data bus decoupling pattern (`tools/data_bus.py` emit/drain) is an elegant solution to a real problem: structured tool output needs to reach the frontend independently of the LLM's narrative stream. The Three.js OmniOrb in `omni-orb.tsx` implements Fibonacci sphere distribution (golden angle `PHI = Math.PI * (3 - Math.sqrt(5))`) for uniform particle density, with per-frame noise-based radial displacement that scales with live audio frequency data from the AnalyserNode. This is the kind of detail that separates genuine engineering effort from glued-together demos.

The primary technical deductions come from two areas. First, the test suite references an API that no longer exists: `test_orchestrator.py` line 89 calls `run_orchestrator("Analyze the Marseille area")` and line 122 calls `_run_agent(mock_agent, "test query")`, but the current `orchestrator.py` only exports `stream_orchestrator`, an async generator. This means a portion of the test suite is testing a prior version of the code and would fail or error on import. Second, the aviation sanction risk feature (`check_aircraft_risk` in `aviation_tools.py`) is placeholder code presented as a real capability.

**Creativity (22/25):** The project's conceptual ambition stands out. Most OSINT tools are geographic — they stop at the Earth's surface. OmniCAT deliberately frames itself as a multi-scale intelligence system, and this framing is coherent from the orchestrator system prompt down to the frontend map-switching logic. The orchestrator prompt contains unusually careful disambiguation instructions: it explicitly forbids geocoding celestial bodies, distinguishes between solar system bodies and exoplanets (with named examples like TRAPPIST-1 vs. Europa), and describes seven cross-domain enrichment patterns the agent should look for. The Vader call easter egg — which triggers the Cyberpunk theme after the video ends via a 10-second timer in `handleVaderVideoEnd` — is a clever demo moment that also serves as a live product demo of the theme system. The synthesized Jarvis-inspired HUD sounds in `sfx.ts` are built from scratch: the `sfxActivate` function layers a sub-bass pulse, a frequency sweep, a filtered noise breath, and three stereo-detuned sine tones in a timed sequence that mimics the Iron Man HUD boot. This level of attention to sensory detail is unusual in hackathon submissions.

**Usefulness (18/25):** The core value proposition — a single conversational interface to cross-reference vessel tracking, aircraft positions, earthquake and wildfire alerts, conflict events, and space weather — is genuinely useful for analysts, journalists, researchers, and security professionals. The geocode-first pipeline (always calling Nominatim before specialists) ensures spatial coherence. The briefing format is designed for voice delivery (max 8-12 sentences, no bullet points, TTS-friendly), which is a real UX constraint the team thought through. However, the practical utility is partially undermined by the mocked aviation risk system and the acknowledgment in `data_bus.py` that the system is not safe for concurrent users. The exoplanet and solar research domains, while impressive technically, have a narrower OSINT audience than the geopolitical and maritime domains.

**Track Alignment (22/25):** Mistral is woven into the architecture at multiple distinct levels: Mistral Large as the reasoning orchestrator, Ministral-3B as the specialist agent backbone, and Mistral Voxtral for real-time speech-to-text over WebSocket. The choice to use different Mistral models for different roles — and to justify that choice (cost/quality tradeoff in `docs/architecture.md`) — demonstrates intentional use of the Mistral API rather than incidental use. AWS Bedrock is the compute provider. The strands-agents framework is an AWS-developed agentic SDK, creating additional Track 1 AWS alignment. The `ORCHESTRATOR_MODEL_ID` and `AGENT_MODEL_ID` environment variables allow overriding the default models, showing the team anticipated model substitution during experimentation. The only uncertainty is whether the Bedrock model IDs (`mistral.mistral-large-3-675b-instruct`, `mistral.ministral-3-14b-instruct`) correspond to actual publicly available Bedrock endpoints, as these naming patterns were not confirmed against current Bedrock documentation.

## Special Challenge Eligibility

The project uses Mistral Voxtral (`voxtral-mini-transcribe-realtime-2602`) for real-time WebSocket speech-to-text, which represents the voice/audio modality of the Mistral API. If there is a voice or multimodal special challenge at this hackathon, OmniCAT is a strong candidate. The ElevenLabs TTS integration with a custom voice built specifically for this project adds further depth to the voice use case. The AWS Bedrock compute usage also qualifies it clearly under the Track 1 AWS sponsorship.
