# Judge Report: vibecraft

**Track:** Track 1 — Anything Goes / AWS
**Repository:** https://github.com/benjamin-feldman/vibecraft

## Summary

Vibecraft is an ambitious project that embeds an AI-powered code editor inside Minecraft, allowing players to describe redstone-logic circuits in natural language and have them compiled to Verilog via a custom Python DSL (mcdsl). The system chains Mistral's text generation (mistral-medium-latest) for DSL code generation, Voxtral for voice-to-DSL transcription, and Ministral-3B for in-editor code explanation with TTS playback, all bridged to a Minecraft Forge mod through a local WebSocket server. The technical scope is remarkable: a fully specified domain-specific language, a multi-stage compiler pipeline, a Verilog backend with macro-ABI, and a polished in-game editor with Vim keybindings and syntax highlighting.

## Strengths

- End-to-end pipeline from natural language (or voice) to executable Minecraft hardware is technically cohesive and fully implemented in code.
- The `mcdsl` DSL is properly designed with a three-type expression system (BoolExpr, IntExpr, EventExpr), a restricted Python front-end with AST validation in `compiler.py`, and a clean IR (`DesignIR`, `MacroInstance`) that separates concerns between compilation and Verilog emission.
- The Verilog backend (`backend.py`) generates well-formed, parameterized module instantiations with blackbox declarations and enforces parameter range constraints (e.g. `TICKS: 1..1200`) — a level of rigour uncommon in hackathon projects.
- The Minecraft Forge mod implements a genuine Vim-modal editor (Normal/Insert modes, `h j k l`, `dd`, `o`, `$`) with a syntax highlighter (`SyntaxHighlighter.java`) and line-number gutter, all rendered in the game's GUI layer.
- Three distinct Mistral models are meaningfully integrated: generation, transcription, and explanation — with TTS audio playback of explanations piped back into the game via `afplay`.
- Six test files cover compiler validation, IR structure, macro ABI constraints, expression semantics, and device runtime, providing real regression coverage.
- The DSL documentation (`AGENTS.md`, `DSL_MACRO_ABI.md`) is detailed enough to guide an LLM to produce correct DSL programs, and the AGENTS.md is injected as system context in every AI call.

## Areas for Improvement

- The MinecraftHDL submodule is referenced but not initialised in the repository; the core compilation target (physical Verilog→redstone) is therefore not demonstrable from a fresh clone without extra steps.
- Server URLs are hardcoded to `brevlab.com` endpoints (e.g. `AUDIO_SERVER_URL = "https://vllm-xxk5iki9.brevlab.com/v1"` in `vibe_server.py`), which will break for any evaluator who cannot reach those servers; these should be environment-variable-only.
- The syntax highlighter (`SyntaxHighlighter.java`) uses C/C++ keywords and types (e.g. `typedef`, `constexpr`, `uint8_t`) rather than Python/DSL keywords, producing incorrect highlighting for the actual DSL files it is intended to display.
- The `vibe_agent.py` launcher delegates all AI work to an external `vibe` binary that must be separately installed; the older agent path is partially superseded by `vibe_server.py` but both code paths remain.
- There is no CI/CD configuration (no GitHub Actions, no Makefile target), making it difficult to verify correctness without a fully configured local environment.
- The README is extremely terse (seven lines) and provides no architecture overview, no description of what the project actually does, and no indication of which Mistral models are used — a significant omission for evaluation and adoption.
- The `prompt_cli.py` uses `client.chat.complete(...)` (Mistral SDK style) while `explain_cli.py` uses `client.chat.completions.create(...)` (OpenAI style), reflecting inconsistent API client usage across the codebase.

## Detailed Reasoning

**Architecture overview.** Vibecraft is structured into three layers: (1) a Python DSL compiler package (`dsl/src/mcdsl`), (2) a Python WebSocket server (`src/vibe_server.py`) that bridges the Minecraft mod to AI services, and (3) a Minecraft Forge mod (`minecraft/vibecraft-mod`) written in Java. The flow is: a player opens the in-game editor on a `.py` DSL file, types or speaks a description, which is sent over WebSocket to the Python server, which invokes the `vibe` agentic coding tool (or directly calls Mistral APIs for explain/audio paths), and the result is compiled by `vibecraft-build` (the `mcdsl.cli:main` entry point) to a Verilog file that the MinecraftHDL mod then places in the world.

**DSL compiler quality.** The `mcdsl` compiler is the strongest technical component. `compiler.py` runs a restricted Python elaboration: it validates the AST before execution (rejecting non-frozen dataclasses, methods, banned imports, mutation of lists), then executes the `main` function in a sandboxed namespace and captures the resulting `OutputRule` objects. The expression system in `expr.py` is correctly typed — `BoolExpr.__bool__` raises `TypeCheckError` to prevent DSL expressions from being used as real Python booleans, and operator overloads propagate type correctly across the `BoolExpr`/`EventExpr`/`IntExpr` hierarchy. The IR (`ir.py`) is fully immutable (frozen dataclasses with slots). The Verilog backend (`backend.py`) correctly scalarises integer inputs into individual 1-bit wires and enforces macro parameter ranges with precise formulas (e.g. `EXPECT_IDX` bound is `(1 << (ceil_log2(BTN_COUNT) * SEQ_LEN)) - 1`).

**Mistral integration.** Three models are integrated with clear separation: `mistral-medium-latest` for structured DSL code generation in `prompt_cli.py` (using tagged output `<explanation>...<code>` to enforce parseable responses), `Voxtral-Mini-3B-2507` for speech transcription in `audio_cli.py` using the `mistral-common` `AudioChunk` protocol directly, and `Ministral-3-14B-Reasoning` for DSL explanation in `explain_cli.py` with an optional TTS synthesis pass through a separate Qwen TTS model. The `mistral_support.py` helper consolidates environment loading and client construction. The use of `mistral-common` for native audio chunk formatting (rather than a generic multipart upload) demonstrates deliberate API knowledge.

**Java Minecraft mod.** The Forge mod is more than a thin wrapper. `EditorScreen.java` implements a proper two-mode (Normal/Insert) modal editor with scrollable viewport, line numbers, cursor rendering (block vs. caret), and syntax-highlighted text via `SyntaxHighlighter.java`. `TextBuffer.java` manages file state with auto-save on every edit and disk-change detection via `checkDiskChange()`. The WebSocket communication classes (`VibeCommandRunner.java`, `AudioRecordRunner.java`, `ExplainRunner.java`) use Java's built-in `java.net.http.WebSocket` API with proper async completion stages and atomic boolean guards for concurrency safety. The F7 key triggers voice recording (hold-to-record), F8 triggers explanation, and the chat box is used for AI response streaming — a coherent UX within Minecraft's constraints.

**Testing.** The test suite (`test_compile_ir.py`, `test_compiler_validation.py`, `test_macro_abi.py`, `test_expr_and_helpers.py`, `test_devices_runtime.py`, `test_scaffold.py`) is substantive. Tests verify both positive cases (correct compilation to IR and Verilog) and negative cases (validation errors for every major constraint: missing `main`, wrong return type, duplicate IDs, output feedback, mutation of collections, profile violations). The macro ABI tests check exact parameter values and port connectivity, including the `EXPECT_IDX` packing arithmetic for `SequenceLock`. This is a genuinely useful test suite, not a superficial one.

**Gaps and risks.** The primary gap is that the MinecraftHDL submodule (the component that actually reads Verilog and places redstone in the world) is not initialised in the repo, so the full end-to-end flow cannot be verified. The hardcoded `brevlab.com` server URLs mean the AI features only work against specific live endpoints. The `vibe_server.py` constructs prompts with a `--max-turns 5` limit on the `vibe` agent, which may be insufficient for complex DSL generation with compilation errors. The README provides essentially no documentation, which is a meaningful omission given the project's complexity.

The project uses Voxtral (speech-to-text) and a custom TTS model for the explain-and-speak feature, suggesting potential eligibility for a multimodal or voice-interface challenge if one exists.
