# Judge Report: shipaleks/mistral-hackaton

**Track:** Track 1 — "Anything Goes" (AWS) — with potential Track 2 (Fine-Tuning) elements
**Repository:** https://github.com/shipaleks/mistral-hackaton

## Summary

Eidetic is an autonomous qualitative research system that conducts adaptive voice interviews via ElevenLabs, processes transcripts with a multi-agent Mistral pipeline (Designer → Analyst → Synthesizer), and evolves its interview script after every conversation based on accumulating evidence. The system implements a simplified version of grounded theory methodology — extracting causal Factor → Mechanism → Outcome propositions, tracking convergence, retroactively scanning all evidence against new hypotheses, and producing a structured research report. It was built as a solo effort during the 48-hour hackathon and is accompanied by a separate optional fine-tuning pipeline using Voxtral for transcription.

## Strengths

- The core pipeline loop (webhook → Analyst → Designer → ElevenLabs PATCH) is fully implemented with idempotency checks, duplicate-conversation deduplication, and graceful fallback at each LLM failure point.
- The `ScriptSafetyGuard` (`services/script_safety.py`) is a thoughtful safety layer that strips personal references ("earlier you mentioned", "you said") from LLM-generated interview prompts before they reach the voice agent, with topic-drift detection using Jaccard overlap on research-question tokens.
- The `SynthesizerAgent` includes a grounding verification step that cross-checks every quoted string in the generated report against the actual evidence store using sequence matching and token overlap, preventing hallucinated quotes from appearing in the output.
- The fine-tuning pipeline (`training/`) is complete end-to-end: Voxtral transcription with chunked fallback for large files, two-role speaker normalization, Mistral Large-based example extraction, PII/Yandex scrubbing with regex + LLM verification, JSONL formatting, and the Mistral fine-tuning job launcher with polling and artifact persistence.
- The visualization backend (`services/visualization.py`) builds a rich hypothesis map — nodes typed as `hypothesis`, `evidence`, and `candidate_cluster`; edges typed as LLM-confirmed or heuristic with opacity/style metadata; weighted Jaccard similarity scoring for unassigned evidence grouping — all served via a single API endpoint with SSE push on each interview.
- 16 test files covering pipeline idempotency, SSE event emission, HMAC signature verification, webhook payload parsing robustness, script safety enforcement, synthesizer grounding rejection, training pipeline stages, and visualization model output.
- The commit history (20 commits across 2 days, Feb 28 – Mar 1) shows consistent forward progress throughout the hackathon with meaningful feature names, not a single late-night dump.
- The PRD and ARCHITECTURE documents are thorough enough to serve as a real product spec, with cost estimates, fallback plans, convergence thresholds, and judging-criteria self-assessment.

## Areas for Improvement

- The described React + d3.js frontend with animated knowledge graph (`ARCHITECTURE.md` mentions `KnowledgeGraph.jsx`, `ConvergenceMeter.jsx`, `ScriptDiff.jsx`, etc.) is replaced by static HTML files (`dashboard/projects.html`, `dashboard/graph.html`) — the demo visualization is functional but significantly simpler than the architecture document promises.
- The LLM client is described as provider-agnostic but only has a Mistral/OpenAI-compatible path; the Anthropic adapter is explicitly deferred ("Anthropic needs adapter, but for hackathon = Mistral only"). The config still lists provider as a field, creating a slight mismatch between design and implementation.
- The analyst system prompt in `prompts/analyst_system.txt` (the file actually used at runtime) is significantly more minimal than the detailed version in `PROMPTS.md`, which contains the full grounded theory guidance. It is unclear whether the runtime behavior fully matches the intended methodology.
- Proposition merge logic (two propositions with >60% evidence overlap) is described in detail in the PRD and PROMPTS.md but there is no `merge` field in the `AnalysisResult` model (only `prunes`); the analyst coercion code does not handle merge payloads. This appears to be a partially implemented feature.
- The fine-tuning pipeline has no actual fine-tuned model artifact or evaluation results demonstrated in the repo — it is a well-built pipeline but a pipeline without demonstrated output at hackathon time.
- There are no integration tests that spin up the full FastAPI server; all tests are unit-level with fakes/mocks. A short e2e smoke test against a real project lifecycle would increase confidence.
- The `training/common.py` `MistralClient` is a second, separate implementation of the HTTP client with its own retry logic, duplicating `agents/llm_client.py`. A shared base client would reduce maintenance surface.

## Detailed Reasoning

**Technicity (22/25).** The codebase is cleanly structured into `agents/`, `services/`, `models/`, `api/`, `training/`, and `prompts/` with clear separation of concerns. The `LLMClient` (`agents/llm_client.py`) handles exponential backoff retries for all transient HTTP errors (408, 429, 500–504) with configurable parameters injected from `config.py`. The `Pipeline` class (`services/pipeline.py`) is the orchestration spine: it saves the interview, runs analyst analysis, applies results atomically to the project state, computes heuristic links, generates a new script with fallback to a minimal version on failure, runs the safety guard, pushes to ElevenLabs, and emits a full suite of SSE events — all in a single async function with proper exception isolation. The webhook handler (`api/routes_webhook.py`) implements HMAC-SHA256 signature verification via `services/webhook_security.py` with timestamp replay protection, and has extensive transcript field normalization to handle ElevenLabs payload variation across API versions. The visualization layer (`services/visualization.py`) is particularly strong: it independently computes evidence similarity, clusters unassigned evidence, generates typed nodes and edges with Jaccard scores, and includes a `potential_supporter` edge type for cross-cluster bridging — this is not boilerplate. The test suite (16 files) includes subtle behavioral tests like `test_pipeline_sanitizes_personalized_prompt_before_sync` and `test_synthesizer_grounding.py` testing the grounding-rejection path. Points are withheld primarily for the missing merged React frontend and the incomplete merge logic.

**Creativity (21/25).** The central insight — that a voice interview loop can implement automated grounded theory with adaptive hypothesis management — is both conceptually sound and practically novel. Existing AI interview tools are static-script chatbots; Eidetic's convergence engine (divergent vs. convergent mode, proposition lifecycle of untested → exploring → confirmed/challenged → saturated → weak/merged) is a thoughtful adaptation of qualitative methodology to an LLM pipeline. The retroactive scan — checking all prior evidence against newly generated propositions — is a clever solution to the ordering problem inherent in iterative hypothesis generation. The cross-lingual proposition graph, where a Japanese-language quote maps to the same English-language proposition as a French one, is architecturally backed by the evidence schema (`quote` stored verbatim in source language, `quote_english` for analysis, `language` ISO code). The `ScriptSafetyGuard` detecting topic drift toward tech-stack discussion (patterns like "your project", "codebase", "implementation") is a pragmatic and creative feature specific to the hackathon research context.

**Usefulness (20/25).** The problem is real and the market is substantial: enterprise qualitative research costs thousands of dollars per study and takes weeks. The system's value proposition is clear — run 10 voice interviews overnight and receive structured, evidence-grounded insights with confidence scores and contradiction tracking. The QR code endpoint for sharing the talk-to link is a practical touch for live event use. The simulate endpoint for text-mode fallback (when ElevenLabs has issues) keeps the core pipeline usable without voice. The report grounding verification prevents hallucinated quotes from reaching stakeholders. The main usefulness concern is that real-world qualitative rigor requires human validation of AI-extracted evidence — the system correctly does not claim to replace a researcher entirely, but the confidence scores are heuristic (supporting / (supporting + contradicting)) rather than statistically grounded, and the proposition merge/subsume logic appears incomplete in the implementation.

**Track Alignment (21/25).** This is a strong Track 1 entry: the Mistral API is central to all intelligence in the system (not just one feature), the ElevenLabs + Mistral Custom LLM integration is genuinely innovative, the deployment target is Koyeb (Mistral's infrastructure acquisition), and the fine-tuning pipeline using Voxtral and the Mistral fine-tuning API shows awareness of the full Mistral ecosystem. The PRD explicitly references W&B for Track 2 cross-submission but the implementation is described as "optional, time-permitting" and no training artifacts are committed. The submission should primarily be evaluated as Track 1.

## Special Challenge Eligibility

**Best Use of ElevenLabs** (stated goal in PRD.md): Strong candidate. The system uses ElevenLabs Conversational AI with Mistral as the Custom LLM backend, implements dynamic post-call webhook processing, and PATCHes the agent system prompt after every interview with a freshly computed script. The voice interview loop is the core product experience, not a peripheral feature.

**Track 2 (Fine-Tuning, W&B)**: Partial eligibility only. The fine-tuning pipeline code is complete but no actual fine-tuned model or training results are demonstrated in the repository. The W&B integration is mentioned in the PRD but not implemented in code.
