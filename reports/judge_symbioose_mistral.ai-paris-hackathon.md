# Judge Report: YouGotIt — Immersive B2B Training Platform

**Track:** Track 1 — Anything Goes (AWS)
**Repository:** https://github.com/Symbioose/Mistral.ai-Paris-Hackathon

## Summary

YouGotIt converts any PDF or TXT training document into a voice-powered, multi-agent RPG simulation in approximately 20 seconds. Employees train by playing through dramatic workplace scenarios, speaking or typing their answers, while HR managers receive a detailed competency heatmap and skills-gap report generated by Mistral. Built solo in 30 hours, the project demonstrates a tight, production-minded full-stack integration of Mistral AI via AWS Bedrock, ElevenLabs TTS, and browser-native Web Speech API.

## Strengths

- Deterministic Q&A state machine: game progression is 100% server-controlled; the LLM only generates dialogue text, eliminating hallucination risk on flow logic. The five-phase machine (`ASKING → REPHRASING → LEARNING → RE_ASKING → COMPLETE`) is cleanly implemented and documented.
- Near-zero latency agent transitions: `prefetchedResponseRef` fires the next agent's API call while the outgoing agent's TTS is still playing, so there is no perceptible gap when the simulation hands off between characters.
- Custom BM25 RAG: `app/lib/rag.ts` implements overlapping chunking, French stop-word removal, Unicode normalization, IDF scoring, and a zero-dependency retrieval function. Each agent's system prompt is populated with chunks most relevant to its role and knowledge topics.
- Parallel orchestration with function calling: `runOrchestration()` in `/api/chat/route.ts` runs a separate Mistral Large call in parallel to the streaming response, using three tools (`update_emotion`, `trigger_event`, `agent_note`) to update agent state, trigger narrative events, and populate inter-agent shared memory — all without blocking the streaming dialogue.
- Comprehensive error handling: every Mistral call has a timeout, explicit fallback path, and graceful degradation. `buildFallbackReport()` and `fallbackGamePlan()` ensure the UI never blocks even if the LLM fails entirely.
- TypeScript discipline: `app/lib/types.ts` defines 20+ interfaces covering the full domain model. All API routes share the same type definitions.
- Voice pipeline detail: emotion states map to specific ElevenLabs parameters (e.g., `panicked` sets `stability: 0.20`, `speed: 1.4`, `style: 0.95`), and stage directions in `*asterisks*` are routed to a separate narrator voice — a subtle but impactful UX decision.
- Model cost optimization: categorization (a simple clustering task) is deliberately routed to `mistral-small` (`magistral-small-2509`) while generation and evaluation use `mistral-large`, showing awareness of cost-performance tradeoffs.

## Areas for Improvement

- The `OPENAI_API_KEY` used for AWS Bedrock expires every 12 hours (noted in `README_TECH.md`), which is a fragile setup for a demo and would require a proper credential refresh mechanism for production.
- `app/page.tsx` is a monolithic 700+ line client component containing state management, TTS pipeline, agent prompt building, SSE consumption, and rendering logic. It would benefit from extraction into custom hooks (`useTtsPipeline`, `useMultiAgentChat`) and smaller components.
- The BM25 RAG is built server-side but the `buildRagIndex` call happens on the client in `page.tsx` (via `ragIndexRef`) — this is an inconsistency that means the index is built twice and the client-side version drives agent prompt assembly. Centralizing RAG to the server would be cleaner.
- The project is in French only, including all system prompts, fallback text, and the STT locale (`fr-FR`). The README acknowledges multi-language support as future work, but the deep hardcoding of French makes that non-trivial to add.
- Three legacy files (`app/lib/agents/orchestrator.ts`, `evaluator.ts`, `agent-factory.ts`) are committed but marked as unused. They should be removed or their reintegration planned to avoid confusing future maintainers.
- The scoring system penalizes with raw point deductions (e.g., `-Math.round(maxPointsPerQuestion * 0.2)`) applied to a 0–100 per-category score, but the score can only increase from zero, which means early failures and later correct answers may not compound cleanly across categories of different sizes.
- No authentication or session isolation is implemented. The landing page mentions "HR tokens" and "team access codes" but these are placeholder UI flows (`landingModalStep: "idle" | "loading" | "done"`) with no backend verification, which would need to be built for production deployment.

## Detailed Reasoning

**Architecture Soundness**

The project's most technically impressive decision is separating what the LLM controls from what the server controls. The Q&A state machine in `/api/chat/route.ts` uses deterministic logic to advance through `InteractionPhase` states — the LLM receives a precisely crafted `agentPrompt` instruction at each step (e.g., "Reformule la question différemment pour l'aider à réfléchir. NE DONNE PAS LA REPONSE, oriente seulement. 25 mots max.") and produces only the dialogue text. This is a mature pattern that avoids the common hackathon failure mode of letting the LLM drive application state.

The `runOrchestration()` function introduces parallel function calling as a side channel. While the primary streaming call produces agent dialogue, a second Mistral Large call resolves `update_emotion`, `trigger_event`, and `agent_note` tool calls. The orchestration result is then merged into the SSE `meta` event sent to the client. The comment "Orchestration is non-critical — fail silently" shows the author understood that this parallel path should never block the core experience.

**Mistral Integration Depth**

Five distinct call patterns are present and each is justified. In `prepare.ts`, the three-step pipeline (`generateQAPairs` → `categorizeQAPairs` → `generateAgentsAndScenario`) correctly uses `response_format: { type: "json_object" }` with a robust `extractJson<T>()` helper that handles array-root JSON, object-root JSON, and malformed output. The `extractJson` function tries direct parse, then regex extraction of the first `{...}` or `[...]` block — a practical defense against LLM preamble. In `/api/chat/route.ts`, two separate Mistral clients are used: `mistralChat()` (via AWS Bedrock) for evaluation and orchestration, and `vercelMistral("mistral-large-latest")` (direct Mistral API) for streaming dialogue. This is not purely redundant — the Vercel AI SDK's `streamText()` provides a cleaner async iterator interface for SSE, while `bedrockClient` is preferred for JSON-mode calls where streaming is unnecessary.

**RAG Implementation**

`app/lib/rag.ts` is a clean, zero-dependency BM25 implementation. Chunking uses overlapping windows (750 chars, 120 char overlap) with soft breaks at newlines. Tokenization strips French stop words and Unicode diacritics. The BM25 formula (`k1=1.2`, `b=0.75`) is standard. `retrieveRelevantChunks()` returns the top-k chunks with positive scores, falling back to the top-k regardless of score if nothing matches. `formatRetrievedContext()` respects a character budget for context injection. The RAG index is used in `page.tsx` to inject per-agent relevant document excerpts into each agent's system prompt at game initialization, which is a meaningful grounding mechanism for domain-specific Q&A.

**Voice Pipeline**

`splitTtsByStageDirections()` in `page.tsx` parses text with `*asterisks*` as stage directions, routes those segments to the `calm_narrator` voice, and routes the rest to the active agent's voice with emotion-adjusted parameters. The TTS pre-fetch pattern (`ttsPreloadRef` storing promises keyed by chunk text) ensures audio starts playing with minimal gap. The retry logic in `/api/tts/route.ts` handles ElevenLabs 409 (concurrent request conflict) and 429 (rate limit) with exponential backoff, and falls back from the streaming endpoint to the standard endpoint — showing real operational experience with the ElevenLabs API.

**Report Generation**

The `/api/report/route.ts` builds a `SimulationReport` with two paths: a `buildFallbackReport()` that derives everything deterministically from game state and conversation history (no LLM required), and a Mistral Large JSON-mode call that augments it. The `sanitizeReport()` function validates every field of the LLM output with explicit type coercion and range clamping, then merges with the fallback. This defense-in-depth approach means the report UI is never blocked regardless of API availability. The report structure itself — `topCriticalGaps`, `failurePatternAnalysis`, `EmployeeVibe`, `actionablePlan7Days`, `decisionTrace` — is genuinely useful for an HR context and goes well beyond a simple score summary.

**Code Quality Notes**

The codebase is notably clean for a solo 30-hour hackathon. Variable naming is consistent, TypeScript generics are used correctly, and comments are meaningful rather than decorative. The `isFeminineName()` heuristic in `prepare.ts` — using French morphological patterns plus an explicit deny-list of male names ending in 'e' — is a charming detail that shows attention to character consistency in the generated scenario. The `sanitizeNarrative()` function correctly preserves `*stage direction*` markup while stripping other markdown, using a temporary placeholder pattern to avoid conflicts with the replacement regex.
