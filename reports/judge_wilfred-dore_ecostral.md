# Judge Report: Ecostral — Frugal AI Optimizer

**Track:** Track 3 — On Device (NVIDIA)
**Repository:** https://github.com/wilfred-dore/Ecostral

## Summary

Ecostral is an AI agent powered by Mistral Large that autonomously finds the most energy-efficient, lowest-carbon, lowest-cost deployment configuration for any HuggingFace model across heterogeneous hardware targets including NVIDIA Jetson Orin. The system runs a closed optimization loop — hardware profiling, W&B memory retrieval, Mistral-driven configuration proposals, real NVML GPU measurements, Pareto frontier selection — and produces hybrid Markdown/LaTeX reports with matplotlib charts. Real GPU results are provided for Mistral-7B-Instruct-v0.3 on an H100 NVL, demonstrating a 36% energy reduction and 50% VRAM reduction at full accuracy retention.

## Strengths

- Real GPU measurements: NVML power polling with trapezoidal integration, not simulated values. The `EnergyMeter` context manager in `ecostral/evaluation/energy.py` is production-quality.
- The closed self-improvement loop (Mistral proposes -> GPU measures -> W&B logs -> Mistral reads past results -> better proposal) is architecturally elegant and directly relevant to both the NVIDIA track and the W&B mini challenge.
- Comprehensive test suite: 8 test files covering Pareto selection, hardware profiling, cost calculation, reasoning engine (with mocked Mistral), optimizer, memory, API, and evaluation — 75 tests passing.
- Honest engineering: the `PRECISION_REDIRECTS` dict in `model_evaluator.py` explicitly redirects FP8 proposals to BF16 with a documented explanation of why torchao's FP8 dispatch is 4x slower on H100 without full TensorRT. The results JSON records `precision_proposed_by_mistral: "fp8"` alongside `precision: "bf16"` and the implementation note. This transparency is rare.
- Multiple real output artifacts committed: `results.json`, `report.md`, `paper.pdf`, and three matplotlib charts for Mistral-7B, TinyLlama (datacenter), TinyLlama (Jetson Orin), and a second Mistral-7B run — evidence the system ran end-to-end on real hardware.
- Thoughtful hardware abstraction: `TARGET_PROFILES` in `targets.py` encodes allowed/forbidden techniques and hardware constraints for all 6 deployment targets; `derive_search_space` correctly gates FP8 on Hopper compute capability, TensorRT on CUDA availability, and tensor parallelism on multi-GPU configs.
- The W&B memory layer (`wandb_mcp.py`) pre-fetches past runs into a formatted prompt context before each Mistral call — a clean implementation of retrieval-augmented optimization.
- Coding-agent skill files (`skills/ecostral-optimizer/SKILL.md`, `skills/wandb-experiment-memory/SKILL.md`) are well-specified and include chaining with official HuggingFace skills, demonstrating ecosystem thinking.

## Areas for Improvement

- The FP8 + TensorRT path requires the NeMo Docker container (`nvcr.io/nvidia/nemo:25.01`) which was not available during the hackathon demo. The most impactful optimization path for H100 is therefore not actually demonstrated end-to-end in the real results — BF16 is the effective best in all committed outputs.
- All 5 iterations of Mistral-7B converge to the same `fp8/tensorrt` proposal from Mistral Large (iterations 1-5 all show `"precision": "fp8", "technique": "tensorrt"` in `results.json`). The self-improvement loop's diversity is limited in practice — Mistral keeps proposing the same config when it is optimal on paper, even though the hardware cannot execute it natively. A novelty penalty or diversity mechanism would strengthen the loop.
- The accuracy benchmark uses MMLU 0-shot log-probability with a small sample (25 samples in the fast demo). For Mistral-7B, the baseline accuracy of 0.57 MMLU is below the model's known capability (~0.64 on full MMLU), suggesting the fast evaluation mode underestimates baseline quality. A more robust accuracy evaluation would improve the scientific validity of the results.
- The FastAPI `/optimize` endpoint uses a stub evaluator (`random.uniform` noise on precision-based constants) rather than the real model evaluator from `model_evaluator.py`. This limits the REST API's production usefulness without additional integration work.
- The WhiteCircle integration is an interesting addition but depends on an invite-only third-party service (`COLUMBIAHACK` invite code), which reduces reproducibility for evaluators and future users.
- The edge target results (Jetson Orin) show latency *increasing* by 76% for INT4 vs FP32, explicitly because H100 proxy measurement does not reflect native Jetson INT4 tensor core behavior. While this is honestly documented, it means the system cannot yet provide a complete end-to-end story for edge deployment without access to real Jetson hardware.

## Detailed Reasoning

**Architecture and code quality.** Ecostral is organized as a proper Python package (`ecostral/`) with 13 submodules, a `setup.py`, and clean separation of concerns: hardware profiling, reasoning, evaluation, optimization, cost calculation, Pareto selection, logging, memory, API, and quality gate are each in their own subpackage. The main orchestrator (`ecostral/loop.py`, 282 lines) is a well-documented `OptimizationLoop` class that accepts an `apply_and_evaluate_fn` callback, making it framework-agnostic. The `CLAUDE.md` file serves as a detailed system spec (16 pages covering frugality philosophy, hardware spec, search strategy, measurement methodology, optimization objective, and the W&B memory schema), which is an unusual and valuable design artifact at a hackathon.

**Mistral integration.** The `ReasoningEngine` in `ecostral/reasoning/mistral_engine.py` calls `mistral-large-latest` with a structured system prompt that constrains the output to a specific JSON schema (`FrugalConfig`), enforces hardware-awareness, and injects the W&B memory context as part of the user message. The `_parse_response` method handles markdown-wrapped JSON, falls back gracefully on parse failure, and sanitizes invalid techniques against the active search space. After optimization, a second Mistral call writes the Executive Summary, Technical Insights, and Recommendations sections of the report, grounded in the real measured numbers — the implementation note in the committed `report.md` for Mistral-7B is an example of Mistral-written prose that correctly cites 38% energy reduction and 50% VRAM reduction.

**Real measurements and scientific validity.** The committed `results.json` files show genuine NVML-measured numbers: Mistral-7B FP32 baseline at 1,399 ms latency, 160.6 μWh energy, 27.0 GB VRAM; BF16 optimized at 1,170 ms, 103.3 μWh, 13.5 GB. The `_NVMLEnergyMeter` class in `model_evaluator.py` uses a 50 ms polling thread with trapezoidal integration — correct methodology for GPU power measurement. The MMLU evaluation uses 0-shot log-probability scoring across 25-100 samples depending on configuration. The `PRECISION_REDIRECTS` mechanism is particularly noteworthy: rather than claiming FP8 performance it cannot achieve, the system transparently redirects FP8 proposals to BF16 and records both the proposed and evaluated precision in `results.json`.

**Pareto optimization.** The `pareto_filter` and `select_best` functions in `ecostral/pareto/selector.py` implement correct multi-objective Pareto dominance (lower or equal on all objectives, strictly lower on at least one) with an accuracy constraint. The final selection uses min-max normalized weighted sum scalarization. The test suite in `tests/test_pareto.py` covers dominated removal, trade-off preservation, accuracy constraint enforcement, empty dataframe handling, and weighted selection — all passing.

**On-Device track alignment.** The Jetson Orin target is a first-class citizen: `targets.py` defines a `JETSON_ORIN` profile with Ampere-compatible techniques (FP16, INT8, INT4, AWQ, GPTQ, TensorRT, KV cache compression), a 64 GB memory ceiling, and max batch size of 4. The `docker-compose.yml` provisions the NeMo 25.01 container for TensorRT-LLM export. The committed `tinyllama_1_1b_chat_v1_0__jetson_orin/results.json` shows INT4 reducing VRAM from 4.1 GB to 0.75 GB — an 81.8% reduction that determines whether TinyLlama fits on a Jetson Orin Nano 8 GB. The README's explicit note that the 700 W H100 cloud call is replaced by 15 W on-device inference (~98% energy saved per device) demonstrates clear understanding of the track's mission.

**W&B self-improvement loop.** The `WandBMemory.get_recent_runs` method queries past runs filtered by deployment target, ordered newest first, and formats them as a tabular prompt context. This context is injected into every Mistral proposal call, enabling the reasoning engine to avoid dominated configurations and propose novel strategies based on empirical results. The committed W&B project links (`optimization-mistral-7b-instruct-v0-3`) provide external evidence that the loop ran and logged data during the hackathon period.
